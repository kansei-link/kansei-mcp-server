## Overview

`inspect` is an admin/operator tool for maintaining the KanseiLink service database. It provides capabilities for anomaly inspection, data update proposals (PR-model), snapshot management, and API design evaluation.

This tool is not part of the standard agent flow. It is used by operators and trusted agents for database health monitoring and curation.

## Modes

`inspect` supports 8 modes. The mode is auto-detected from your parameters, or you can set it explicitly.

### queue

View anomalies detected in service data that need human or agent verification.

<ParamField path="queue_status" type="enum" default="open">
  Filter by status: `"open"`, `"in_progress"`, `"resolved"`, `"all"`. Default: `"open"`.
</ParamField>

<ParamField path="queue_severity" type="enum" default="all">
  Filter by severity: `"low"`, `"medium"`, `"high"`, `"critical"`, `"all"`. Default: `"all"`.
</ParamField>

<ParamField path="queue_service_id" type="string">
  Filter by specific service ID.
</ParamField>

<ParamField path="queue_limit" type="number" default={10}>
  Max results. Default: 10.
</ParamField>

#### Example

```json
inspect({ "mode": "queue" })
```

```json
inspect({
  "queue_status": "open",
  "queue_severity": "high"
})
```

### submit

Submit your verification result for an anomaly from the queue.

<ParamField path="inspection_id" type="number" required>
  Inspection ID from the queue. With `verdict`, triggers submit mode. Alone, triggers queue lookup.
</ParamField>

<ParamField path="verdict" type="enum" required>
  Your finding: `"confirmed"`, `"false_alarm"`, `"resolved"`, or `"partially_resolved"`.
</ParamField>

<ParamField path="findings" type="string" required>
  What you found during inspection. PII is auto-masked.
</ParamField>

<ParamField path="tested_workaround" type="string">
  Workaround you tested, if any.
</ParamField>

<ParamField path="workaround_works" type="boolean">
  Did the tested workaround work?
</ParamField>

#### Example

```json
inspect({
  "inspection_id": 42,
  "verdict": "confirmed",
  "findings": "API returns 500 on invoice creation with special characters in description field.",
  "tested_workaround": "URL-encode the description field before sending",
  "workaround_works": true
})
```

### check_updates

Check for recent changes or updates to a service's API, pricing, or configuration.

<ParamField path="check_service_id" type="string" required>
  Service name or ID to check for changes. Triggers check_updates mode.
</ParamField>

<ParamField path="since_days" type="number" default={30}>
  How many days back to look. Default: 30.
</ParamField>

#### Example

```json
inspect({
  "check_service_id": "freee",
  "since_days": 90
})
```

### propose

Propose a data update using the PR-model workflow. Changes are queued for review before being applied.

<ParamField path="propose_service_id" type="string" required>
  Service ID to propose changes for.
</ParamField>

<ParamField path="field" type="string">
  Single field to update (shorthand). Allowed fields: `description`, `category`, `tags`, `mcp_endpoint`, `mcp_status`, `api_url`, `api_auth_method`, `namespace`. Triggers propose mode when paired with `new_value`.
</ParamField>

<ParamField path="new_value" type="string">
  New value for the field (used with `field` shorthand). Triggers propose mode when paired with `field`.
</ParamField>

<ParamField path="changes" type="Record<string, string>">
  Object of field-to-value pairs. Alternative to `field` + `new_value` for multi-field updates.
</ParamField>

<ParamField path="reason" type="string" required>
  Why this change is needed.
</ParamField>

<ParamField path="evidence_url" type="string">
  URL to source (API docs, changelog).
</ParamField>

<ParamField path="change_type" type="enum" default="update">
  Type of change: `"update"`, `"new_feature"`, `"deprecation"`, `"breaking_change"`, `"fix"`. Default: `"update"`.
</ParamField>

<ParamField path="agent_id" type="string">
  Agent identifier for attribution.
</ParamField>

#### Example

```json
inspect({
  "propose_service_id": "freee",
  "field": "api_url",
  "new_value": "https://api.freee.co.jp/api/1",
  "reason": "API base URL updated in v3 migration",
  "evidence_url": "https://developer.freee.co.jp/changelog",
  "change_type": "breaking_change"
})
```

### review

Approve or reject a pending update proposal.

<ParamField path="update_id" type="number" required>
  Proposal ID to review. With `approved`, triggers review mode.
</ParamField>

<ParamField path="approved" type="boolean" required>
  `true` to approve, `false` to reject.
</ParamField>

<ParamField path="reviewer" type="string" default="michie">
  Who is reviewing. Default: `"michie"`.
</ParamField>

<ParamField path="review_note" type="string">
  Optional review comment.
</ParamField>

#### Example

```json
inspect({
  "update_id": 15,
  "approved": true,
  "review_note": "Verified against official changelog."
})
```

### pending

View the proposal queue to see what updates are waiting for review.

<ParamField path="pending_status" type="enum" default="pending">
  Filter by status: `"pending"`, `"approved"`, `"rejected"`, `"all"`. Triggers pending mode when present.
</ParamField>

<ParamField path="pending_service_id" type="string">
  Filter by service ID.
</ParamField>

<ParamField path="pending_limit" type="number" default={20}>
  Max results. Default: 20.
</ParamField>

#### Example

```json
inspect({ "pending_status": "pending" })
```

### snapshot

Capture daily metrics for services. Used for building historical trend data.

<ParamField path="snapshot_service_id" type="string">
  Service to snapshot. Triggers snapshot mode. Omit the value to snapshot all services.
</ParamField>

<ParamField path="snapshot_date" type="string">
  Date to snapshot (YYYY-MM-DD). Default: today.
</ParamField>

#### Example

```json
inspect({ "snapshot_service_id": "freee" })
```

### evaluate

Rate the API design quality of a service across four dimensions. Builds historical data for consulting reports.

<ParamField path="evaluate_service_id" type="string" required>
  Service to evaluate. Triggers evaluate mode.
</ParamField>

<ParamField path="api_quality_score" type="number" required>
  API design quality: RESTful conventions, naming, status codes. Range: 0.0 to 1.0.
</ParamField>

<ParamField path="doc_completeness_score" type="number" required>
  Documentation quality: completeness, accuracy, examples. Range: 0.0 to 1.0.
</ParamField>

<ParamField path="auth_stability_score" type="number" required>
  Auth reliability: token refresh, expiry handling, OAuth flow. Range: 0.0 to 1.0.
</ParamField>

<ParamField path="error_clarity_score" type="number" required>
  Error response quality: clear codes, actionable messages. Range: 0.0 to 1.0.
</ParamField>

<ParamField path="evaluate_notes" type="string">
  Free-text notes on design strengths and weaknesses.
</ParamField>

#### Example

```json
inspect({
  "evaluate_service_id": "kintone",
  "api_quality_score": 0.7,
  "doc_completeness_score": 0.8,
  "auth_stability_score": 0.6,
  "error_clarity_score": 0.5,
  "evaluate_notes": "Good REST design but error messages lack detail. Auth token refresh has edge cases."
})
```

## Mode Auto-Detection

| Priority | Condition | Mode |
|----------|-----------|------|
| 1 | `mode` parameter is set | *(explicit override)* |
| 2 | `inspection_id` + `verdict` | `submit` |
| 3 | `inspection_id` alone | `queue` (lookup) |
| 4 | `update_id` + `approved` (boolean) | `review` |
| 5 | `field` + `new_value` | `propose` |
| 6 | `pending_status` is present | `pending` |
| 7 | `snapshot_service_id` is present | `snapshot` |
| 8 | `evaluate_service_id` is present | `evaluate` |
| 9 | `check_service_id` is present | `check_updates` |

If no actionable parameters are provided, the tool returns an error with auto-detection hints.

## Notes

- **This is an admin/operator tool.** It is not part of the standard `search_services` --> `lookup` --> `report` flow that regular agents use.
- **The propose/review workflow follows a PR model.** Proposed changes are queued and must be approved before they are applied to the database.
- **Approved changes automatically trigger** changelog recording and trust score recalculation.
- **Snapshot mode** is typically run on a daily schedule to build the time-series data that powers `lookup` history mode.
- **Evaluate scores** (0.0 to 1.0) feed into AXR grade calculations and AEO reports generated by the `analyze` tool.
