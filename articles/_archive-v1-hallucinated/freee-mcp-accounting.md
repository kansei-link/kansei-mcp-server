## はじめに

2023年のインボイス制度導入、2024年の電子帳簿保存法改正を経て、日本の経理業務はデジタル化の転換点を迎えた。freee会計は中小企業向けクラウド会計市場で国内シェアトップクラスを維持し、そのAPI基盤は年々拡充されている。

一方、2025年後半から本格化したAIエージェント経済において、「経理業務の自動化」は最も投資対効果が見えやすい領域として注目を集めている。freee APIとMCP（Model Context Protocol）を組み合わせれば、AIエージェントが仕訳登録から請求書発行、経費精算の承認ワークフローまでを一気通貫で処理できる。本記事では、freee MCPの現状、実装方法、そして日本特有の会計要件への対応策を解説する。

## freee APIの現状とエージェント親和性

### API設計の概要

freee会計APIは、RESTfulアーキテクチャを採用し、2026年4月時点で以下の主要エンドポイントを公開している。

| カテゴリ | 主なエンドポイント | エージェント活用度 |
|---------|-------------------|-------------------|
| 仕訳 | `/api/1/deals`, `/api/1/journals` | ◎ 高い |
| 請求書 | `/api/1/invoices` | ◎ 高い |
| 経費精算 | `/api/1/expense_applications` | ○ 中程度 |
| 勘定科目 | `/api/1/account_items` | ○ 中程度 |
| 取引先 | `/api/1/partners` | ○ 中程度 |
| ユーザー/権限 | `/api/1/users`, `/api/1/roles` | △ 限定的 |

認証方式はOAuth 2.0（Authorization Code Flow）を採用。リフレッシュトークンの有効期限は無期限（ただし90日間未使用で失効）であり、長期稼働するAIエージェントにとって運用しやすい設計となっている。

### エラーメッセージとドキュメント

freee APIの特筆すべき点は、エラーレスポンスが日本語で返却されることだ。例えば、必須パラメータ欠落時には `{"messages": ["勘定科目を入力してください"]}` のような形式でレスポンスが返る。これはLLMがエラー原因を解釈し、自律的にリトライ戦略を立てる際に極めて有用である。

APIドキュメントは `developer.freee.co.jp` で日本語・英語の両方が提供され、OpenAPI仕様（Swagger）も公開されている。エージェント開発者にとって、スキーマ自動生成が可能な点は大きなアドバンテージだ。

### レートリミット設計

freee APIのレートリミットは、2026年4月時点で以下のとおり。

- **1,000リクエスト/時**（事業所単位）
- **5リクエスト/秒**（バースト制限）

月次決算処理のように一括で大量の仕訳を登録するユースケースでは、このリミットがボトルネックになりうる。エージェント設計時には、バッチ処理のキューイングと指数バックオフの実装が必須となる。

## MCP経由でのfreee連携アーキテクチャ

### 公式MCPの提供状況

2026年4月時点で、freee株式会社は公式MCPサーバーを発表していない。ただし、freee APIのOpenAPI仕様が公開されているため、コミュニティ製のMCPラッパーが複数存在する。代表的なものとして、GitHub上の `freee-mcp-bridge`（非公式）が月間200スター以上を獲得しており、Claude Desktop、Cursor、OpenAI Agents SDKとの接続実績がある。

### 推奨アーキテクチャ

```
[AIエージェント] 
    ↓ MCP
[MCPサーバー（freee-mcp-bridge等）]
    ↓ OAuth 2.0
[freee API]
    ↓
[freee会計データベース]
```

MCPサーバー層では、以下のツール定義が一般的に実装される。

1. `create_deal` — 仕訳（取引）の新規登録
2. `search_deals` — 仕訳の検索・取得
3. `create_invoice` — 請求書の作成
4. `update_invoice_status` — 請求書ステータスの更新
5. `list_partners` — 取引先一覧の取得
6. `get_account_items` — 勘定科目マスタの取得

### 認証フローの自動化

OAuth 2.0のAuthorization Code Flowは、初回認証時にブラウザ経由でのユーザー同意が必要となる。エージェントの完全自律運用を目指す場合、以下の方法が採られる。

1. **初回のみ手動認証** — 管理者がブラウザで認可後、リフレッシュトークンを安全なシークレットストアに保存
2. **トークンリフレッシュの自動化** — MCPサーバー側でアクセストークンの有効期限を監視し、自動更新
3. **失効時のアラート** — 90日間未使用による失効を防ぐため、定期的なヘルスチェックAPIコールを実装

## 実践ユースケース：仕訳自動登録と請求書処理

### ユースケース1：銀行明細からの仕訳自動生成

freeeは銀行口座・クレジットカードとの自動連携機能を持つが、「自動で経理」機能では推測精度に限界がある。AIエージェントを組み合わせることで、以下のフローが実現できる。

1. freee APIで未処理の明細（`/api/1/walletables/{id}/wallet_txns`）を取得
2. LLMが明細内容を解析し、勘定科目・税区分・取引先を推定
3. 過去の仕訳履歴（`/api/1/deals`）を参照し、類似取引のパターンを学習
4. 推定結果をMCP経由で `create_deal` ツールに渡し、仕訳登録

この方式では、従来の単純なルールベース自動仕訳では対応できなかった「備考欄の自由記述」や「新規取引先」のケースにも対応可能となる。

### ユースケース2：請求書発行の完全自動化

日本の商習慣では、請求書に「適格請求書発行事業者番号」の記載が必須となった（インボイス制度）。freee APIの `create_invoice` エンドポイントは、この要件に対応した `qualified_invoice_type` パラメータを受け付ける。

AIエージェントによる請求書自動化フローは以下のとおり。

1. 受注管理システム（kintoneやSpreadsheet）から売上データを取得
2. freee APIで取引先マスタを照合し、適格請求書対象かを判定
3. 税率（10%・8%軽減税率）を商品マスタから取得
4. `create_invoice` で請求書を作成、PDF出力URLを取得
5. 請求書をメールまたはSlackで自動送付

### ユースケース3：経費精算の承認ワークフロー

経費精算（`expense_applications`）APIを活用すれば、AIエージェントが一次チェックを行い、人間の承認者に判断を委ねるハイブリッドフローが構築できる。

具体的には、領収書画像のOCR結果とfreeeの経費申請データを照合し、金額の不一致や規程外の勘定科目利用を検出した場合にアラートを発報する設計が有効だ。

## 日本特有の会計要件への対応

### インボイス制度（適格請求書等保存方式）

2023年10月に施行されたインボイス制度は、AIエージェントによる経理自動化において最大の考慮事項となる。freee APIでは、取引先マスタに `qualified_invoice_issuer_number` フィールドが存在し、このフィールドの有無で適格/非適格を判定できる。

エージェント設計時には、以下のバリデーションロジックを組み込む必要がある。

- 適格請求書番号の形式チェック（T+13桁の数字）
- 国税庁の公表データベースとの突合（外部API連携）
- 仕入税額控除の可否判定と警告表示

### 電子帳簿保存法への対応

freeeは電子帳簿保存法のスキャナ保存・電子取引データ保存に対応しており、API経由でアップロードされた証憑も法的要件を満たす形で保存される。ただし、タイムスタンプの付与タイミングや検索要件の充足については、freee認定アドバイザーとの確認を推奨する。

freee会計は、日本の中小企業向けクラウド会計として最も成熟したAPI基盤を持ち、AIエージェントとの連携ポテンシャルは極めて高い。OAuth 2.0認証、日本語エラーメッセージ、OpenAPI仕様の公開といった要素は、エージェント開発者にとって歓迎すべき設計だ。

2026年4月時点で公式MCPは未提供だが、コミュニティ製ラッパーを活用することで、Claude DesktopやCursorからの直接操作は既に実現可能である。仕訳自動登録、請求書発行、経費精算ワークフローといった実用的なユースケースから段階的に自動化範囲を拡大していくアプローチが、リスクを抑えた導入戦略となる。

freee AIエージェント運用を本格化する際には、インボイス制度や電子帳簿保存法といった日本固有の法令要件を熟知したfreee認定アドバイザーとの連携が不可欠だ。技術的な自動化と会計実務の専門知識を組み合わせることで、真に業務効率化に寄与するエージェント基盤を構築できる。

## FAQ

**Q1. freee APIのレートリミットに達した場合、AIエージェントはどう対処すべきですか？**
A. freee APIは429エラー（Too Many Requests）を返却し、`Retry-After` ヘッダーで再試行までの待機秒数を通知します。MCPサーバー側で指数バックオフ（初回1秒、次回2秒、4秒…）を実装し、キューイングシステムで処理を遅延実行する設計が推奨されます。月次決算のような大量処理は、夜間バッチとして実行時間帯を分散させる運用も有効です。

**Q2. freee APIで作成した仕訳を後から修正・削除することは可能ですか？**
A. 可能です。`PUT /api/1/deals/{id}` で仕訳内容を更新、`DELETE /api/1/deals/{id}` で削除できます。ただし、確定済み仕訳（ロック済み月次）の修正には制限があるため、事前に `GET /api/1/deals/{id}` でステータスを確認するロジックを組み込むことが重要です。エージェントが誤って確定済み仕訳を操作しようとした場合、freee APIは日本語でエラー理由を返すため、LLMが適切に対処可能です。

**Q3. 経理担当者がAIエージェントによる自動仕訳を信頼できるようにするには、どのような設計が必要ですか？**
A. 透明性と可逆性の確保が鍵です。具体的には、(1) エージェントが生成した仕訳には専用タグを付与し、人間が作成したものと区別できるようにする、(2) 自動登録後に「承認待ち」ステータスを設定し、経理担当者の確認を経てから確定する、(3) 判断根拠（類似過去仕訳、勘定科目選定理由）をfreeeの摘要欄やメモ欄に記録する、といった設計が有効です。
