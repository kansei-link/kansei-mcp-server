# AEOスコアを上げる5つの方法 — SaaS企業のための実践ガイド

**著者:** KanseiLink編集部

## TL;DR（この記事の要点）

- AEOスコア改善の第一歩は、APIエラーメッセージの「エージェント可読性」を高めることにある
- MCP対応は差別化要因から必須要件へ移行中——2026年Q2時点で対応済みの国内SaaSはまだ少数
- ドキュメントの日英併記と構造化データ（JSON-LD/OpenAPI）がエージェント経由の発見可能性を左右する
- レートリミット設計は「人間向け」から「エージェント向け」へ再設計が必要
- 日本固有の業務ドメイン（請求書・年末調整・押印フロー）をAPIで表現できるかが国内SaaSの競争力になる

## はじめに

2026年、AIエージェントが業務SaaSを直接操作する時代が本格化している。Claude、GPT、Geminiをはじめとする大規模言語モデルは、MCP（Model Context Protocol）やFunction Callingを通じて外部サービスと連携し、人間の指示を受けてタスクを自律実行する。この文脈で急速に注目を集めているのが「AEO（Agent Engine Optimization）」——AIエージェントから選ばれ、正しく呼び出されるためのSaaS側の最適化である。

従来のSEOが「検索エンジンに見つけてもらう」技術だったとすれば、AEOは「AIエージェントに使ってもらう」技術だ。日本のSaaS企業にとって、AEOスコアの改善は単なる技術トレンドではなく、エージェント経済における生存戦略となりつつある。本記事では、SaaS企業がAEOスコアを上げるための具体的な5つの方法を、実在するサービスの状況を踏まえながら解説する。

---

## 1. APIエラーメッセージを「エージェント可読」に再設計する

AEO最適化の第一歩として見落とされがちなのが、APIエラーメッセージの品質である。人間の開発者は曖昧なエラーメッセージでもドキュメントを参照して原因を推測できるが、AIエージェントはエラーレスポンスの文字列から次のアクションを判断する必要がある。

### 悪い例と良い例

```json
// 悪い例：エージェントが原因を特定できない
{"error": "Bad Request", "code": 400}

// 良い例：原因と推奨アクションが明示されている
{
  "error": {
    "type": "validation_error",
    "message": "The 'invoice_date' field must be in ISO 8601 format (YYYY-MM-DD).",
    "field": "invoice_date",
    "suggestion": "Convert the date to ISO 8601 format and retry."
  }
}
```

freeeやSmartHRのAPIは、2026年4月時点でエラーメッセージの構造化が比較的進んでおり、エラータイプ・対象フィールド・推奨アクションを含む形式を採用している。一方、一部の国内SaaSでは依然として「エラーが発生しました」のような人間向けの曖昧なメッセージが返却されるケースがある。

### チェックポイント

- エラーレスポンスに`error_type`、`field`、`suggestion`または`next_action`フィールドがあるか
- HTTPステータスコードとエラー内容が一致しているか（422 Validation Errorを400で返していないか）
- 日本語エラーメッセージに英語の`error_code`が併記されているか

## 2. MCP対応を段階的に進める

MCP（Model Context Protocol）は、AnthropicがClaudeエコシステム向けに公開したオープンプロトコルで、AIエージェントと外部ツール・データソースの接続を標準化する。2026年4月時点で、Notion、Slack、GitHub、StripeなどグローバルSaaSの多くが公式または準公式のMCPサーバーを提供している。

### 日本SaaSのMCP対応状況（2026年4月時点）

| サービス | MCP対応状況 | 備考 |
|---------|------------|------|
| kintone | コミュニティ製MCPサーバーが存在 | Cybozu公式は未発表 |
| freee | 2026年Q1に公式MCPサーバーをベータ公開 | 会計・人事労務の一部エンドポイント対応 |
| SmartHR | 2026年4月時点で公式MCPは未発表 | REST APIは充実 |
| Chatwork | 公式MCPは未発表 | Webhook連携は可能 |
| Backlog | 公式MCPは未発表 | コミュニティ製が開発中 |

MCP対応はSaaS AIエージェント対応の象徴的な施策だが、段階的なアプローチが現実的だ。

### MCP対応のステップ

1. **既存REST APIの棚卸し**：どのエンドポイントがエージェント操作に適しているか評価
2. **OpenAPI仕様の整備**：MCPツール定義に変換しやすい状態を作る
3. **認証方式の検討**：OAuth 2.0のリフレッシュトークン運用がエージェント向けに適切か確認
4. **コミュニティMCPのサポート**：公式対応前でもコミュニティ製MCPサーバーへのAPIアクセスを容認する姿勢を示す

## 3. ドキュメントを構造化データで補強する

AIエージェントがSaaSの機能を「発見」するプロセスは、人間がドキュメントを読むプロセスとは異なる。エージェントはOpenAPI仕様、JSON-LDによる構造化データ、そしてMCPツール定義を優先的に参照する。

### ドキュメント整備の優先度

1. **OpenAPI 3.1仕様の完全な記述**：全エンドポイント、パラメータ、レスポンススキーマ、エラーコードを網羅
2. **operationIdの命名規則統一**：`createInvoice`、`getEmployeeById`など、エージェントが推論しやすい動詞+名詞形式
3. **descriptionフィールドの充実**：各エンドポイントの目的・前提条件・副作用を200字程度で記述
4. **examplesの充実**：リクエスト・レスポンスの具体例を日本の業務文脈で提供（請求書の例なら税率10%・8%の混在など）

### 日本語と英語の併記

日本のSaaS固有の課題として、ドキュメントが日本語のみで提供されているケースがある。グローバルなAIエージェント（Claude、GPTなど）は日本語を理解するが、英語で記述されたAPI仕様のほうが推論精度が高い傾向がある。AEO改善を目指すなら、少なくともOpenAPI仕様とエラーメッセージは日英併記を検討すべきだ。

## 4. レートリミットを「エージェント時代」向けに再設計する

AIエージェントは人間のユーザーと異なる呼び出しパターンを持つ。1つのタスク遂行のために複数のエンドポイントを連続で呼び出す「チェーン実行」が頻発し、従来の「1ユーザーあたりN回/分」というレートリミット設計では摩擦が生じる。

### エージェント対応のレートリミット設計原則

- **バースト許容**：短時間の連続呼び出しを一定数まで許容する
- **ヘッダーでの残量通知**：`X-RateLimit-Remaining`、`X-RateLimit-Reset`を全レスポンスに含める
- **429レスポンスにRetry-Afterを明示**：エージェントが自動リトライのタイミングを判断できるようにする
- **エージェント専用プラン**：API呼び出し回数ベースの従量課金プランを用意し、人間ユーザーとは別枠で管理する

Stripeは2025年後半から「Agent API Access」という専用のレートリミット枠を提供しており、これをモデルに国内SaaSも追随する動きが予想される。

## 5. 日本固有の業務ドメインをAPI化する

グローバルSaaSとの差別化において、日本の業務慣習に根ざした機能のAPI化は最大の武器になる。

### 日本固有ドメインの例

- **適格請求書（インボイス制度）対応**：登録番号の検証、税率ごとの内訳取得
- **年末調整・法定調書**：扶養控除申告書、源泉徴収票のデータ取得・生成
- **電子帳簿保存法対応**：タイムスタンプ付与状況、検索要件充足の確認
- **押印・承認フロー**：稟議状態の取得、承認者のエスカレーション

これらの業務フローをAPIで完結できるSaaSは、AIエージェントから「日本の業務処理において代替不可能なツール」として選好される。freeeの確定申告API、SmartHRの年末調整関連エンドポイントは、こうした日本固有ドメインのAPI化の先行事例といえる。

エージェント対応を進める際は、単に「CRUDを提供する」だけでなく、業務フローを抽象化した「高レベルAPI」（例：`POST /complete-year-end-adjustment`）の提供も検討すべきだ。これにより、エージェントは複数のAPIを組み合わせる推論負荷を軽減でき、成功率が向上する。

## まとめ

AEOスコアを上げるための5つの方法を整理すると、その本質は「APIをエージェントの立場で再設計する」ことに集約される。エラーメッセージ、MCP対応、ドキュメント、レートリミット、ドメイン特化API——いずれもエージェントが「このSaaSを使いたい、使える」と判断するための要素だ。

日本のSaaS企業にとって、英語圏SaaSへのキャッチアップと日本固有の業務ドメインでの差別化を両立させる必要がある。特にMCP対応は2026年後半にかけて「あれば差別化」から「なければ不利」へと急速にシフトすることが予想される。公式対応が難しくても、コミュニティ製MCPサーバーの開発を阻害しない姿勢、OpenAPI仕様の公開、エージェントフレンドリーなレートリミット設計など、今日から着手できる施策は多い。

AEO最適化は一度やって終わりではない。エージェントの進化、プロトコルのアップデート、ユーザー企業の運用パターン変化に応じて継続的に改善を重ねる必要がある。本記事で紹介した5つの方法を起点に、自社SaaSの「エージェント体験」を定期的に評価・改善していただきたい。

## FAQ

**Q1. AEOスコアを計測する公式ツールはありますか？**
A. 2026年4月時点で、AEOスコアの業界統一基準や公式計測ツールは存在しない。KanseiLINKが独自に公開しているAEOランキングは、MCP対応状況、APIドキュメントの構造化度、エラーメッセージ品質、レートリミット設計、日本ドメイン対応の5軸で評価している。自社サービスの現状把握には、まずOpenAPI仕様の完全性チェック、エージェントによる実際のAPI呼び出しテストを推奨する。

**Q2. 小規模SaaSでもAEO対応に取り組むべきですか？**
A. 取り組む価値はある。MCP対応やドキュメント整備は大企業だけの課題ではなく、むしろ小規模SaaSのほうがアジリティを活かして先行できる領域だ。特にニッチな業務ドメイン（特定業界の基幹業務など）に特化したSaaSは、その領域で「エージェントが頼る唯一のAPI」になれる可能性がある。まずはOpenAPI仕様の整備とエラーメッセージの改善から始めることを勧める。

**Q3. 日本語のみのAPIドキュメントはエージェント対応で不利ですか？**
A. 不利になる場面がある。Claude、GPT、Geminiなど主要なLLMは日本語を理解するが、内部の推論プロセスは英語ベースで訓練されているため、英語で記述されたAPI仕様のほうが意図解釈の精度が高い傾向がある。全ドキュメントの英訳が難しければ、OpenAPI仕様のdescriptionフィールドと主要なエラーメッセージだけでも日英併記にすることで、エージェント経由の発見可能性と操作成功率が向上する。
