# Chatwork vs Slack — AIエージェント対応力を徹底比較

**著者:** KanseiLink編集部

## TL;DR（この記事の要点）

- Slack MCPは2025年後半に公式リリースされ、エージェント連携の成熟度で先行している
- Chatwork APIは日本語ドキュメントと国内サポートに強みがあるが、公式MCPは2026年4月時点で未発表
- API設計思想の違いにより、エラーハンドリングとレートリミットの扱いで差が生じる
- 日本企業の業務フローを自動化する場合、ChatworkとSlackの選択はエージェント運用コストに直結する
- 両ツール併用環境ではブリッジ設計とMCPサーバーの二重管理が課題となる

## はじめに

2026年、日本のSaaS市場においてAIエージェントの導入が本格化している。バックオフィス業務の自動化から営業支援まで、エージェントが人間に代わってSaaSを操作するシーンが急増した。この文脈で注目されるのが「ビジネスチャットのAI対応力」である。

日本国内で圧倒的なシェアを持つChatworkと、グローバルスタンダードとして浸透したSlack。この二大ビジネスチャットをAIエージェント視点で比較すると、API品質、MCP対応状況、ドキュメント整備度に明確な差が見える。本記事では、エージェント運用担当者・SaaS開発者が実務で判断できるレベルまで両者の違いを掘り下げる。

---

## MCP対応状況の現在地

### Slack MCPの先行優位

Slackは2025年9月、Model Context Protocol（MCP）に正式対応した公式MCPサーバーをリリースした。これによりClaude、GPT系エージェント、その他MCP対応AIから、Slackのチャンネル操作・メッセージ送受信・ファイル管理が標準化されたプロトコルで可能になっている。

Slack MCPの特徴は以下の通りである：

- OAuth 2.0ベースの認証フローがMCPツール定義に組み込まれている
- チャンネル一覧取得、メッセージ投稿、スレッド返信、リアクション追加などの主要操作が標準ツールとして提供される
- エラーレスポンスがMCP仕様に準拠しており、エージェントが再試行判断を行いやすい

### Chatwork MCPの現状

一方、Chatwork MCPは2026年4月時点で公式サーバーが未発表である。Chatwork社はAPI v2を提供しており、REST APIとしての基本機能は整備されているが、MCPラッパーはコミュニティ実装に依存している状況だ。

KanseiLINK編集部が確認した限り、GitHub上にはChatwork APIをMCPサーバー化した非公式実装が複数存在する。ただし、これらは以下の課題を抱えている：

- 認証トークンの管理がMCP標準の認証フローに統合されていない
- エラーメッセージのマッピングが実装者依存で一貫性がない
- Chatwork固有の「マイチャット」「概要欄」などの概念がツール定義に反映されていないケースがある

## Chatwork APIとSlack API——設計思想の比較

### エンドポイント設計とリソースモデル

Slack APIはリソース指向で設計されており、`conversations.*`、`chat.*`、`files.*`といった名前空間で機能が整理されている。これはエージェントにとって「何ができるか」を発見しやすい構造である。

Chatwork APIもREST原則に従っているが、エンドポイント数がSlackと比較して限定的だ。メッセージ検索やスレッド単位の操作など、Slackでは標準的な機能がChatworkでは提供されていない、または制限がある。

### 認証方式の違い

| 項目 | Chatwork API | Slack API |
|------|--------------|-----------|
| 主要認証方式 | APIトークン（固定） | OAuth 2.0 |
| トークンスコープ | 全権限 or なし | 細粒度スコープ指定可 |
| トークンローテーション | 手動 | 自動更新対応 |
| Bot専用トークン | 非対応 | 対応（Bot Token） |

この差はエージェント運用において重要である。Slackではエージェントに「メッセージ投稿のみ」「特定チャンネルのみ」といった最小権限を付与できる。Chatworkでは現状、APIトークンが全権限を持つため、エージェントの暴走リスクを技術的に制限しづらい。

### エラーメッセージとレートリミット

Slack APIのエラーレスポンスは`error`フィールドに機械可読なエラーコード、`response_metadata`に詳細情報を含む。エージェントはこれを解析して「権限不足なら管理者に通知」「レートリミットなら待機後リトライ」といった分岐を実装できる。

Chatwork APIもHTTPステータスコードとエラーメッセージを返すが、エラーコードの種類がSlackほど細分化されていない。また、レートリミット到達時の`Retry-After`ヘッダー対応など、エージェントが自律的に対処するための情報が限定的である。

## 日本企業のエージェント運用における実務的考慮

### 日本語ドキュメントとサポート体制

Chatwork APIの日本語ドキュメントは完全日本語で提供されており、国内企業のエンジニアにとってアクセスしやすい。サポート窓口も日本語対応であり、API利用に関する問い合わせへの応答速度は良好との声が多い。

Slack APIのドキュメントは基本的に英語である。日本語翻訳ページも存在するが、更新が遅れるケースがあり、最新機能は英語ドキュメントを参照する必要がある。ただし、Slack MCPの導入ドキュメントはMCP公式サイト経由でアクセス可能であり、英語に抵抗がなければ問題ない。

### 既存ワークフローとの統合

日本企業の多くはChatworkを「社内外連絡の中心」として使用しており、取引先とのやり取りもChatwork上で完結している。このため、エージェントがChatworkを操作できることは実務上のインパクトが大きい。

一方、エンジニア組織やスタートアップではSlackが主流である。GitHub、Notion、Jira等の開発ツールとの公式連携が充実しており、エージェントがこれらを横断操作する際のハブとしてSlackが機能しやすい。

### 両ツール併用環境の課題

大企業や複数拠点を持つ組織では、部門ごとにChatworkとSlackが併用されているケースがある。この環境でエージェントを運用する場合、以下の課題が生じる：

- MCP対応状況が異なるため、エージェントのツールセットを二重管理する必要がある
- 同一ユーザーの識別がチャット間で統一されていない
- メッセージのブリッジ（転送）処理に独自実装が必要

## AEO観点でのスコアリング指標

KanseiLINKでは、SaaSのAIエージェント対応力を評価するAEOスコアを独自に定義している。Chatwork APIとSlack APIを主要指標で比較した結果は以下の通りである（スコアは編集部による定性評価）。

| 評価指標 | Chatwork | Slack |
|----------|----------|-------|
| 公式MCP対応 | 未対応 | 対応済 |
| APIドキュメント完成度 | 高（日本語完備） | 高（英語中心） |
| 認証の柔軟性 | 低 | 高 |
| エラーハンドリング品質 | 中 | 高 |
| レートリミット設計 | 中 | 高 |
| 日本市場適合性 | 高 | 中 |

この表はあくまで2026年4月時点の状況を反映したものであり、Chatworkが公式MCPをリリースすれば評価は変動する可能性がある。

## まとめ

ChatworkとSlackのAIエージェント対応力を比較すると、MCP対応と認証設計においてSlackが先行していることは明らかである。Slack MCPは公式サポートにより安定したエージェント連携が可能であり、グローバル基準のセキュリティモデルを備えている。

一方、Chatworkは日本市場において根強い利用基盤を持ち、取引先連絡や社内コミュニケーションの中核を担っている。公式MCPのリリースが待たれるところだが、現時点でもChatwork APIを活用したコミュニティMCPサーバーでエージェント連携を試行することは可能である。

エージェント運用担当者へのアクションとしては、まず自社のビジネスチャット利用状況を棚卸しし、Slackメインの場合は公式MCPの導入を検討、Chatworkメインの場合は非公式MCPサーバーの検証または公式対応の情報収集を継続することを推奨する。両ツール併用環境では、ブリッジ処理のコストを見積もった上でエージェント設計を行うべきである。

## FAQ

**Q1. Chatwork MCPは今後公式リリースされる予定はあるか？**
A. 2026年4月時点でChatwork社からMCPに関する公式アナウンスは確認されていない。ただし、Chatwork社はAPI機能拡充を継続しており、AIエージェント対応への関心を示す発言はイベント等で見られる。公式ブログやプレスリリースを定期的にチェックし、リリース時に迅速に対応できる体制を整えておくことを推奨する。

**Q2. SlackとChatworkを併用している場合、エージェントはどちらを優先すべきか？**
A. 業務フローにおける「意思決定の起点」がどちらかを基準にすべきである。例えば、顧客対応がChatwork、開発タスク管理がSlackであれば、顧客対応自動化にはChatworkエージェント、開発ワークフロー自動化にはSlackエージェントを構築する。無理に一本化せず、用途ごとに最適なチャネルを選択する設計が現実的である。

**Q3. Chatwork APIのレートリミットはエージェント運用に支障があるか？**
A. Chatwork APIのレートリミットは5分あたり100リクエスト程度とされており、通常の業務自動化では十分な場合が多い。ただし、大量メッセージの一括処理やリアルタイム監視を行う場合は制限に抵触する可能性がある。エージェント設計時にはリトライロジックと処理キューイングを組み込み、レートリミットを超えない運用設計を行うべきである。
